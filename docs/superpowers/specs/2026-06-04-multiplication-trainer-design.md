# Тренажёр таблицы умножения — дизайн

Дата: 2026-06-04
Статус: одобрено

## Цель

Умный персональный тренажёр таблицы умножения для ребёнка. Не просто random-тест:
запоминает результаты по каждой карточке и чаще показывает то, что знают хуже или
давно не повторяли. До 10 пользователей. Приоритет — надёжность и простота.

## Стек

- **Next.js 15 (App Router) + React + TypeScript** — единое full-stack приложение
  (SPA-фронтенд + API-роуты в одном проекте, без отдельного бэкенда).
- **Prisma ORM + SQLite** локально. Переезд на Postgres = смена `datasource`, код не меняется.
- **Tailwind CSS** — адаптивный, чистый UI.
- **bcryptjs** для хэша пароля; подписанная HTTP-only cookie для сессии (без внешних провайдеров).
- **Vitest** для unit-тестов (алгоритм, скоринг), **Playwright** опционально для e2e (вне MVP).

## Структура приложения

- `/login` — форма входа (имя + пароль).
- `/` — основной SPA-экран, три состояния без перезагрузки: **Settings → Training → Summary**.
- API-роуты (server-side, авторитетные):
  - `POST /api/login`, `POST /api/logout`
  - `POST /api/session/start` → создаёт `Session`, возвращает `sessionId` + первую карточку
  - `POST /api/session/answer` → принимает ответ, валидирует, считает score, обновляет
    статистику, **возвращает следующую карточку**
  - `POST /api/session/finish` → возвращает summary

## Поток данных (одна заявка на ответ)

1. Клиент рисует карточку, **таймер крутится на клиенте** (плавно, без задержек сети).
2. На ответ клиент шлёт `{ sessionId, cardId, shownA, shownB, userAnswer, responseTimeMs }`.
3. Сервер сам определяет правильность, `isAfterTimeout` (по `responseTimeMs` vs лимит сессии),
   `score` (100/50/0), пишет `Attempt`, обновляет `UserCardStats`, **выбирает следующую карточку**
   и возвращает её.
4. Клиент следит за временем сессии. Когда оно вышло — текущий вопрос доигрывается, затем
   клиент зовёт `finish`.

**Скоринг и выбор карточек — только на сервере** (клиент не может схитрить). Тайминг меряется на
клиенте — для детского тренажёра это приемлемо.

## Модель данных

Канонизация карточек: храним только пары `a ≤ b`, `2 ≤ a,b ≤ 9` → ровно **36 карточек**.
`3×8` и `8×3` — одна карточка (одна статистика, один анти-повтор). При показе случайно
переворачиваем порядок; в `Attempt` пишем фактически показанные `shownA/shownB`.

### User
- `id`, `name` (unique), `passwordHash`, `createdAt`

### Card
- `id`, `a`, `b` (a ≤ b), `answer`. Уникальный индекс `(a, b)`.

### UserCardStats
- `userId`, `cardId` (составной PK)
- `attemptsCount`, `fastCorrectCount`, `slowCorrectCount`, `wrongCount`
- `averageScore` (running mean всех попыток, 0..1)
- `recentAverageScore` (EMA, α≈0.3, 0..1) — аппроксимация «последних 5–10 попыток», O(1)
- `averageResponseTimeMs`
- `lastAskedAt`, `updatedAt`

### Attempt
- `id`, `userId`, `cardId`, `shownA`, `shownB`, `correctAnswer`, `userAnswer`,
  `isCorrect`, `isAfterTimeout`, `score` (0/0.5/1), `responseTimeMs`, `sessionId`, `createdAt`

### Session
- `id`, `userId`, `startedAt`, `finishedAt`, `answerTimeLimitSec`, `trainingDurationMin`,
  `totalAnswers`, `averageScore`, `fastCorrectCount`, `slowCorrectCount`, `wrongCount`

## Скоринг

- Правильно до таймера → `score = 1.0` (100%), `fastCorrect`.
- Правильно после таймера → `score = 0.5` (50%), `slowCorrect`.
- Неправильно → `score = 0.0` (0%), `wrong`.
- После истечения таймера вопрос **не** завершается автоматически — пользователь ещё может ответить.
- **Первый пример** сессии — без активного таймера: `isAfterTimeout = false` всегда, правильный → 100%.

## Алгоритм выбора следующей карточки

```
priority = (1 - recentAverageScore) * 60
         + overdueScore * 25
         + newCardScore * 30        // если attemptsCount < 3
         + random(0..10)
```

- `recentAverageScore`: EMA из `UserCardStats` (новая карточка → считается как 0, т.е. максимальный вклад).
- `newCardScore`: даётся, если `attemptsCount < 3` (масштабируется по близости к 3).
- `overdueScore = min(1, daysSinceLastAsked / targetIntervalDays)`.
- `targetIntervalDays` по уровню знания (по `recentAverageScore`): плохо=1, средне=3, хорошо=7, отлично=14.
- **Анти-повтор**: исключаем канонические карточки из последних 3–5 ответов сессии. Если после
  фильтра пусто (мало карточек) — берём из всех.
- Выбор: грузим все ≤36 `UserCardStats` пользователя одним запросом, считаем приоритеты в коде,
  weighted-random по приоритетам. Пропорции 20/50/20/10 получаются естественно из весов.

Алгоритм выделен в чистую функцию `pickNextCard(cards, stats, recentCardIds, now, rng)` —
тестируется детерминированно через инъекцию `rng` и `now`.

## Авторизация

- Несколько пользователей, у каждого свой прогресс. Вход = имя + пароль (bcrypt-хэш).
- Сессия — подписанная HTTP-only cookie с `userId` (HMAC, секрет из env).
- Начальные пользователи создаются seed-скриптом (дефолтный с паролем `12345`).
  Регистрации в UI нет (YAGNI для ≤10 человек) — добавить пользователя = строка в seed.

## Завершение тренировки

Идёт выбранное число минут; текущий вопрос доигрывается. Summary:
всего ответов, быстрых правильных, правильных после таймера, ошибок, средний score,
самые слабые карточки, давно не повторявшиеся карточки.

## UI / дизайн

- Светлый фон, крупная типографика, большая карточка с примером.
- Тренировочный экран: сверху история последних 3–5 ответов (зелёный/оранжевый/красный,
  для ошибки — «4 × 6 = 48 ✗ правильный: 24»); в центре крупный пример `7 × 8 = [input]`;
  слева круговой таймер; справа поле ввода (сразу в фокусе); снизу прогресс/статистика сессии.
- После ответа — плавная подсветка и авто-переход к следующему примеру.
- Цвета: синий (текущий вопрос), зелёный (успех), оранжевый (медленный правильный), красный (ошибка).
- Адаптивно: desktop / tablet / mobile.

## Деплой (позже)

Локально: `npm run dev` + SQLite-файл. Публикация — Vercel + Neon Postgres
(меняем `datasource` в Prisma, гоним миграции). Архитектура готова к переезду с первого дня.

## Тестирование

- Unit (Vitest): чистые функции скоринга (`computeScore`) и выбора карточки (`pickNextCard`),
  канонизация карточек, обновление статистики (`updateStats` как чистая функция над текущими полями).
- Эти модули спроектированы без зависимости от БД/сети → быстрые детерминированные тесты.
- API-роуты тонкие: парсинг запроса → вызов чистой логики → запись в Prisma.
